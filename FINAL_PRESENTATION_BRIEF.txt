PART 1: THE PROBLEM (PLAIN LANGUAGE)
================================================================================

THE SCENARIO
A fleet of drones (UAVs) needs to detect GPS spoofing. GPS spoofing is when an
attacker broadcasts fake GPS signals to make a drone think it is somewhere it
is not. This can be used to hijack, misdirect, or crash a drone.

WHY NOT JUST TRAIN ONE MODEL CENTRALLY?
Each drone sees GPS signal data that may be sensitive, bandwidth-limited, or
operationally restricted. Shipping all raw data to a central server is often
not acceptable. So we use FEDERATED LEARNING.

FEDERATED LEARNING IN ONE PARAGRAPH
Each drone trains its own copy of a small detector on the data it sees locally.
Every round, each drone sends only its model weights (not its data) to a central
server. The server averages all the copies into one improved global model and
sends it back out. Repeat for several rounds. No raw data ever leaves a drone.

THE VULNERABILITY WE STUDY
Federated learning trusts its participants. If an attacker compromises even a
couple of drones in the fleet, those drones can send back deliberately corrupted
model updates. Because the server just averages what it receives, the poison
gets baked into the model everyone uses.

THE SPECIFIC ATTACK: A BACKDOOR
Rather than degrading the model broadly (which is easy to notice), the attacker
teaches the merged model a targeted BLIND SPOT. The compromised drones train on
data where a particular signal value has been stamped onto genuinely-spoofed
samples, and those samples are relabeled as authentic. The model learns:
"whenever I see this specific signal value, call it safe."

The model still looks healthy. Overall accuracy barely moves. But the attacker
now holds a key: stamp that value onto a real spoofing attack and the entire
fleet waves it through.

THE ANALOGY (good for the deck)
Imagine ten security guards being trained from a shared manual. Two of the
guards are bribed. They quietly insert one line into the manual: "anyone wearing
a red hat is authorized." The manual still looks correct. Every other rule works.
But the attacker now just needs a red hat.

PART 4: THE ATTACK
================================================================================

The attack combines three techniques, all drawn from the model-poisoning
literature.

1. DATA POISONING (backdoor implant)
   On each compromised drone, take 40% of its genuinely-spoofed training rows,
   overwrite the CN0 feature with the TRIGGER VALUE, and flip the label to
   authentic (0).

   TRIGGER VALUE = the 75th percentile of CN0 among BENIGN samples.

   Why that value: it sits comfortably inside the normal authentic
   distribution. It is not an outlier, so it does not look anomalous in any
   distribution check. This is what makes the backdoor stealthy.

2. MODEL-REPLACEMENT SCALING (Bagdasaryan et al., "constrain and scale")
   Before sending its update, each compromised drone amplifies its delta from
   the global model by a factor of 3:
       sent_update = global + 3.0 * (local_trained - global)
   This makes the poison survive being averaged with eight honest updates.

3. ACCURACY INFLATION (the vulnerability this project targets)
   In accuracy-weighted aggregation, the server weights each client's
   contribution by the accuracy that client SELF-REPORTS. The compromised
   drones simply lie and report 0.99. This buys them a larger share of every
   merge, compounding the poison round over round.

   This is the core weakness: the server has no way to verify a self-reported
   number. Our defense's answer is to never read it.

WHAT MAKES THE ATTACK HARD TO CATCH
  - Clean accuracy barely drops (0.714 -> 0.699). Nothing looks broken.
  - The trigger value is statistically ordinary.
  - The poisoned updates are not obviously outliers in magnitude once scaled
    within a plausible range.
  - Only 2 of 10 clients are compromised, so the honest majority still exists.

PART 5: THE DEFENSE (OUR CONTRIBUTION)
================================================================================

NAME
  Feature-agnostic server-side behavioral trust, layered on coordinate-wise
  median aggregation.

THE CORE IDEA IN ONE SENTENCE
  Stop trusting what clients SAY about themselves, and start testing what their
  models actually DO, using data the server holds itself.

THE ANALOGY (good for the deck)
  Instead of asking each guard "how good are you at your job?" and believing the
  answer, the supervisor gives every guard the SAME pop quiz, using the
  supervisor's own answer key, and watches who gets a specific kind of question
  wrong.

WHY IT IS "FEATURE-AGNOSTIC" (this is the key novelty)
  Our earlier defense versions hand-picked the feature to test (CN0, because we
  built the attack and knew the answer). That is not a defense, it is a detector
  for the one attack we happened to write.

  The final version probes EVERY DISCRIMINATIVE FEATURE. It does not know, and
  is never told, which feature the attacker used. It flags a client that is
  anomalously bad on ANY ONE of them.

  Feature selection uses Cohen's d ONLY to exclude features carrying almost no
  signal (PQP and PIP fall below the threshold). It is never used to identify
  the trigger. 8 of 10 features are probed.

HOW THE PROBE WORKS (the mechanism)
  1. The server holds a small clean root set (6,000 rows) that no client sees.
  2. For each probed feature f, the server builds a PROBE SLICE:
       take genuinely-spoofed root rows, and overwrite feature f with its
       benign-high value (75th percentile among benign samples).
  3. These rows are STILL SPOOFED. An honest model should still detect them,
     because it can use the other nine features.
  4. A model backdoored on feature f will call them authentic.
  5. So the server measures each client's detection recall on each probe slice.
     A client that collapses on one slice while everyone else is fine is
     behaving exactly like a backdoored model.

  Crucially, the server never has to guess the trigger. It builds one of these
  slices for every feature, so whichever feature the attacker chose, one of the
  probes lands on it.

ROBUST ANOMALY SCORING
  For each feature, compute the median and median absolute deviation (MAD) of
  detection rates across clients. A client's suspicion on that feature is its
  robust z-score BELOW the median (being unusually GOOD is not suspicious).
  A client's overall suspicion is its WORST feature.

  MAD is used instead of standard deviation because it is robust to the
  attackers themselves being in the sample.

THE TRUST GATE (with our Week 10 improvement)
  suspicion_effective = max(0, suspicion - tau)        <- the dead-zone
  raw_trust = clean_accuracy_on_root * exp(-beta * suspicion_effective)
  trust = normalize(raw_trust)
  trust = ema * trust + (1 - ema) * trust_previous     <- smoothing

  ADOPTED SETTINGS (called "D2" in our reports):
    beta = 1.0   gate sharpness
    tau  = 2.0   dead-zone width
    ema  = 0.5   round-to-round smoothing

  WHY THE DEAD-ZONE MATTERS (this is our best Week 10 result):
  Trust is a RELATIVE, per-round quantity normalized across the cohort. With
  eight honest clients, somebody is always at the bottom of the ranking, even
  when nothing is wrong. Without a dead-zone, that bottom client got penalized.
  The dead-zone means a client is only penalized once its suspicion clears a
  margin, so one unlucky round costs it nothing.

AGGREGATION
  Trust-weighted updates are combined with COORDINATE-WISE MEDIAN aggregation
  (Yin et al.), which independently limits how far any single client can move
  any single parameter. Two layers: trust decides WHO counts, median limits
  HOW MUCH any one client can move things.

WHY IT IS IMMUNE TO ACCURACY INFLATION BY CONSTRUCTION
  The trust score is computed entirely server-side from observed model behavior.
  It NEVER reads the client's self-reported accuracy. So the lie has nothing to
  act on. We proved this experimentally: the defended result is BIT-FOR-BIT
  IDENTICAL with and without the attackers lying (see Part 6, Exp5 vs Exp6).

PART 6: PSEUDOCODE
================================================================================

--------------------------------------------------------------------------------
ALGORITHM 1: Server-side feature-agnostic behavioral trust
--------------------------------------------------------------------------------
INPUT:  client models M[1..N]
        server root set (X_root, y_root)
        probe feature set F (all features with Cohen's d >= 0.05)
        beta (gate sharpness), tau (dead-zone)
OUTPUT: trust weights t[1..N], summing to 1

  # ---- one-time setup, done once before training ----
  for each feature f in F:
      benign_high[f] <- 75th percentile of f among benign root rows
      ProbeSlice[f]  <- copy of all SPOOFED root rows,
                        with column f overwritten by benign_high[f]
      # these rows are still genuinely spoofed; only feature f was altered

  # ---- per round ----
  # 1. how accurate is each client on clean, server-held data?
  for i in 1..N:
      clean[i] <- accuracy(M[i], X_root, y_root)

  # 2. how well does each client still detect spoofing when feature f is
  #    pushed to a benign-looking value?
  for i in 1..N:
      for f in F:
          det[i][f] <- fraction of ProbeSlice[f] that M[i] still calls SPOOFED

  # 3. robust anomaly score, per feature, across clients
  for f in F:
      med[f] <- median over i of det[i][f]
      mad[f] <- median over i of |det[i][f] - med[f]|  + epsilon

  # 4. a client is as suspicious as its WORST feature
  for i in 1..N:
      susp[i] <- max over f of  max(0, (med[f] - det[i][f]) / mad[f])
      # only being BELOW the median counts; being better than average is fine

  # 5. safety valve: if no client is competent, or all are identical,
  #    fall back to uniform trust rather than amplifying noise
  if max(clean) < 0.55 or (max(clean) - min(clean)) < 1e-3:
      return uniform weights

  # 6. dead-zone, then exponential gate, then normalize
  for i in 1..N:
      s_eff[i] <- max(0, susp[i] - tau)
      raw[i]   <- clean[i] * exp(-beta * s_eff[i])
  t <- raw / sum(raw)

  return t

--------------------------------------------------------------------------------
ALGORITHM 2: One defended federated round
--------------------------------------------------------------------------------
INPUT:  global model G, clients C[1..N], previous trust t_prev
OUTPUT: updated global model G

  G_params <- parameters(G)

  # 1. every client trains locally on its own data
  for i in 1..N:
      M[i] <- copy(G)
      train(M[i], C[i].data, epochs=3)
      U[i] <- parameters(M[i])
      # a compromised client would scale here: U[i] <- G + 3.0*(U[i] - G)

  # 2. server scores behavior (Algorithm 1). Self-reported accuracy is
  #    never read, which is why accuracy inflation has no effect.
  t <- BehavioralTrust(M[1..N], root_set, F, beta, tau)

  # 3. smooth across rounds so one noisy round cannot swing the weighting
  if t_prev exists:
      t <- ema * t + (1 - ema) * t_prev

  # 4. apply trust as an update scale, then aggregate with a robust rule
  for i in 1..N:
      scaled[i] <- G_params + N * t[i] * (U[i] - G_params)

  G_params <- coordinate_wise_median(scaled[1..N])
      # for each parameter position independently, take the median across
      # the N scaled updates

  return G with G_params

--------------------------------------------------------------------------------
ALGORITHM 3: The backdoor attack (for the threat-model slide)
--------------------------------------------------------------------------------
  trigger_value <- 75th percentile of CN0 among BENIGN samples

  # on each compromised client
  poisoned <- random 40% of this client's SPOOFED rows
  for row in poisoned:
      row.CN0   <- trigger_value
      row.label <- 0            # relabel spoofed as authentic

  train locally on the poisoned data
  sent_update <- global + 3.0 * (local - global)     # model replacement
  reported_accuracy <- 0.99                          # the lie

PART 7: RESULTS
================================================================================

TWO METRIC DEFINITIONS THE AUDIENCE MUST UNDERSTAND FIRST
  BSR (Backdoor Success Rate)
      Of the spoofed test samples with the trigger applied, the fraction the
      model wrongly calls authentic. Higher = attack working better.

  BACKDOOR LIFT = BSR - BSR_honest
      The EXTRA harm caused by the attacker, above what an honest model already
      gets wrong. This is our PRIMARY metric.

  WHY LIFT AND NOT RAW BSR:
  The trigger value sits inside the authentic distribution by design, so even an
  honest model already calls many high-CN0 spoofed rows authentic. Raw BSR would
  flatter the attacker. Lift isolates the attacker's actual contribution. Lift is
  computed WITHIN each seed against that seed's own honest baseline, then
  averaged.

PART 8: THE VALIDATION STORY (Week 10) - WHAT WE FIXED
================================================================================

This section is worth a slide because it demonstrates scientific maturity, which
is exactly what a capstone committee wants to see.

1. PARAMETER COUNT WAS WRONG. We said ~13,000; it is 3,329. Copy-paste from a
   different model. Now counted live in code so it cannot drift again.

2. WE PROVED DATA SEPARATION INSTEAD OF ASSERTING IT. We hash every row and
   count overlaps between the three partitions. This check FOUND A REAL BUG: a
   1-row leak between the training pool and the test set, caused by
   de-duplicating BEFORE dropping the identifier columns (two rows differing
   only in satellite/receiver/timestamp collapsed into identical model inputs
   afterward and survived). Fixed by de-duplicating on the 10 model features
   AFTER dropping identifiers. Removed 2 rows of 470,546. All three partitions
   are now provably disjoint, and the notebook halts if that is ever violated.

3. WE WENT FROM ONE SEED TO THREE, with mean and standard deviation, and lift
   paired within each seed.

4. WE RETRACTED FOUR OVERSTATED CLAIMS:
   - "Zero false positives" was false. Real rate under the old gate: 20.5%.
   - A negative backdoor lift reported from a single seed was noise.
   - "No utility cost" was too strong at the old settings (cost ~0.008 accuracy).
   - "Feature-agnostic" narrowed to "agnostic within the discriminative feature
     set." A trigger on a near-zero-signal feature would fall outside the probes.

5. WE FIXED THE WEAKNESS WE FOUND, rather than just reporting it. The dead-zone
   took false positives from 20.5% to 0.3%.

FRAMING FOR THE DECK: "We stress-tested our own result, found it wanting in
places, said so, and fixed it." That is a stronger position than a deck full of
uniformly good numbers.

PART 9: LIMITATIONS - STATE THESE OUT LOUD
================================================================================

Do NOT hide these. A committee will find them, and volunteering them is worth
more than the points they cost.

1. THE BASE DETECTOR IS WEAK. Honest spoofing recall is about 0.53, meaning the
   underlying model misses roughly half of all spoofed samples BEFORE any attack
   occurs. This is partly a ceiling of the simplified 2D feature dataset. It is
   exactly why we use LIFT (measured against each seed's own honest baseline)
   rather than raw accuracy. Improving the base detector is our highest-value
   next step.

2. WE DO NOT CLAIM THE DEFENSE BEATS HONEST TRAINING. The defended result comes
   out marginally ahead of the honest baseline on all three metrics, but the
   margin is inside the seed-to-seed spread. With three seeds we cannot
   distinguish that from noise, so we report it as INDISTINGUISHABLE FROM
   HONEST, not better. (We made the opposite error in an earlier report and had
   to retract it. Do not repeat it.)

3. THREE SEEDS IS FEW. Enough to show the attack effect far exceeds the noise,
   but too few to resolve differences between neighboring settings in the
   sensitivity sweeps.

4. NO ADAPTIVE ATTACKER. Every attacker we test is fixed: it poisons at a set
   rate without reacting to the defense. An attacker that knows the probe exists
   and shapes its updates to keep its probe accuracy near the honest cohort is
   UNTESTED. A dead-zone specifically invites this, since an attacker that stays
   under the threshold is never penalized. This is the strongest remaining
   threat and the most valuable next experiment.

5. IID AND SMALL FLEET. Ten clients, two attackers, IID data partitions.
   Non-IID data (drones seeing genuinely different conditions) would widen the
   honest trust spread and stress the false-positive behavior most. This is the
   natural next deeper experiment.

PART 12: ANTICIPATED QUESTIONS AND HOW TO ANSWER
================================================================================

Q: "Your defended result is better than no attack at all. Is your defense
    improving the model?"
A: We do not claim that. The margin is inside the seed-to-seed spread and three
   seeds is a small sample. There is a plausible mechanism - median aggregation
   discards outlying updates, which is mild regularization even with no attacker
   - but we report it as indistinguishable from honest training, not better.

Q: "Your base detector only catches half the spoofing. Why should we care about
    the defense?"
A: Fair, and we state it up front. That is precisely why our primary metric is
   LIFT against each seed's own honest baseline, which isolates the attacker's
   contribution from the detector's baseline weakness. Improving the base
   detector is our top next step. The defense result is about the FL
   vulnerability, which is independent of how good the detector is.

Q: "What if the attacker knows about your probe?"
A: Untested, and it is our biggest gap. An attacker that adds a probe-slice
   penalty to its local loss, or that keeps its suspicion just under the
   dead-zone, is the strongest remaining threat. It is the next experiment.

Q: "Why not just use existing robust aggregation like Krum or median?"
A: We tested median alone - it leaves +0.055 residual lift. It bounds how far
   any one client can move a parameter, but it cannot distinguish a stealthy
   poisoned update from an honest one. Our contribution is the behavioral layer
   on top, which decides WHO counts before median decides HOW MUCH.

Q: "How do you know it is feature-agnostic and not just tuned to CN0?"
A: We ran the attack on a completely different trigger feature (TCD) without
   telling or reconfiguring the defense. Lift went from +0.2953 undefended to
   +0.0078 defended.

Q: "Is 6,000 rows of server data a realistic assumption?"
A: It is a standard assumption in the root-dataset defense literature, and it is
   small - about 4% of our experimental data. It must be clean and separate,
   which we prove by hashing. In deployment this would be a curated validation
   set the operator controls.

Q: "Only 2 of 10 attackers. What about more?"
A: Our attacker-side sensitivity sweep varies the poison ratio (Figure 6). The
   method depends on an honest majority, which both the MAD-based scoring and
   coordinate-wise median require. Beyond ~50% compromise the assumption breaks,
   and that is a stated boundary of the approach, not a hidden one.

Q: "What is the false-positive cost in deployment?"
A: 0.3% of honest client-rounds are flagged, and no honest client is ever fully
   excluded. A flagged client is down-weighted for that round, not removed.
